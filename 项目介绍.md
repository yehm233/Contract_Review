# Contract-Review-Pro V1.4 项目介绍

## 一、项目概述

Contract-Review-Pro 是一个面向企业法务场景的**智能合同审查系统**。用户通过 OnlyOffice 文档编辑器的侧边栏插件提交合同，系统自动完成实体抽取、风险扫描、替代条款生成，并将结构化审查结果返回前端，支持一键采纳将建议条款替换进文档。

系统基于 **LangGraph Agent + MCP（Model Context Protocol）** 架构构建，核心特点是将 AI 的"决策能力"与"执行能力"彻底分离——Agent 负责编排决策，MCP 工具负责具体分析，两者通过标准协议通信。

### 核心能力

- 合同实体自动抽取（当事人、金额、管辖、生效日期等）
- 多维度法律风险扫描（付款、违约、管辖、知识产权、保密、不可抗力等）
- 针对性替代条款生成（不是模板，而是根据具体风险点生成）
- OnlyOffice 文档内一键采纳建议条款
- 完整的审查链路追踪与日志

## 三、MCP 协议与工具层

### 3.1 什么是 MCP

MCP（Model Context Protocol）是一个开放协议，定义了 AI 模型与外部工具之间的标准通信方式。协议基于 JSON-RPC 2.0，传输层支持 stdio 和 HTTP SSE。

本项目中 MCP 的角色：

```text
AI Agent（决策者）
    │
    │  MCP 协议：tools/list（发现工具）、tools/call（调用工具）
    │
    ▼
MCP Server（执行者）
    │
    │  内部调用 LLM 完成具体分析
    │
    ▼
返回结构化结果给 Agent
```

### 3.2 MCP Server 端

**入口**：[mcp_server/main.py](mcp_server/main.py)

```python
from app.server import mcp
if __name__ == "__main__":
    mcp.run(transport="sse")  # SSE 传输，监听 :8000
```

**工具注册**：[mcp_server/app/server.py](mcp_server/app/server.py)

通过 `@mcp.tool()` 装饰器注册 5 个工具，FastMCP 框架自动完成：
- 工具名、描述、参数 JSON Schema 的声明
- `tools/list` 请求的响应
- `tools/call` 请求的路由和参数反序列化

### 3.3 MCP 工具清单

| 工具 | 输入 | 功能 | LLM Prompt 角色 |
|---|---|---|---|
| `extract_contract_entities` | `text: str` | 提取甲方、乙方、金额、管辖、生效日期、标的 | 合同实体抽取助手 |
| `risk_assessment_scan` | `text: str` | 逐条扫描合同，识别所有法律风险点 | 中国法律顾问 |
| `generate_revision_clause` | `risk_type: str`, `original_text?: str` | 针对风险类型生成具体替代条款 | 合同条款起草专家 |
| `draft_contract_from_template` | `form_data: dict` | 根据合同要素生成完整草稿 | 合同起草助手 |
| `format_to_legal_standard` | `text: str` | 将非正式文本改写为正式法律中文 | 法律文书润色专家 |

### 3.4 工具内部 LLM 调用

每个工具通过 [mcp_server/app/llm.py](mcp_server/app/llm.py) 调用 LLM：

```python
async def call_llm(system: str, user: str) -> str:
    resp = await client.chat.completions.create(
        model=MODEL,                                    # 从环境变量读取
        messages=[
            {"role": "system", "content": system},      # 专业角色 prompt
            {"role": "user", "content": user},          # 合同文本
        ],
        temperature=0,
    )
    return resp.choices[0].message.content
```

**降级策略**：每个工具都有 `try/except`，LLM 调用失败时返回关键词匹配结果或模板文本，不会向上抛异常。

### 3.5 MCP 客户端（后端侧）

后端通过 `langchain-mcp-adapters` 的 `MultiServerMCPClient` 连接 MCP Server：

```python
# langchain_review.py: _load_tools()
self._client = MultiServerMCPClient({
    "contract_review": {
        "transport": "sse",
        "url": MCP_SSE_URL,  # 默认 http://127.0.0.1:8000/sse
    }
})
self._tools = await self._client.get_tools()  # 自动完成 tools/list 发现
```

`get_tools()` 内部自动完成 MCP 协议握手：
1. 建立 SSE 连接
2. 发送 `tools/list` 请求
3. 接收工具定义（name、description、inputSchema）
4. 转换为 LangChain 兼容的 Tool 对象

## 四、Agent 工作流

### 4.1 两层 LLM 架构

本项目在 Agent 模式下存在**两层 LLM 调用**，这是理解架构的关键：

```text
第一层 LLM（后端 Agent）：
  ChatOpenAI(gpt-4.1-mini)
  职责：理解合同 → 决定调用哪些工具 → 综合工具结果 → 输出结构化结论
  位置：langchain_review.py → agent.ainvoke()

第二层 LLM（MCP 工具内部）：
  AsyncOpenAI(gpt-4.1-mini)
  职责：执行具体分析任务（提取实体、扫描风险、生成条款）
  位置：mcp_server/app/server.py → call_llm()
```

第一层是**决策者**（编排），第二层是**执行者**（分析）。两者解耦，可以独立替换模型或优化 prompt。

### 4.2 Agent 创建

[langchain_review.py:158-179](backend/app/services/langchain_review.py#L158-L179)

```python
async def _load_agent(self):
    tools = await self._load_tools()          # MCP 工具发现
    model = self._build_chat_model()          # ChatOpenAI(model=gpt-4.1-mini, T=0)
    self._agent = create_agent(
        model=model,
        tools=tools,                          # MCP 工具列表
        system_prompt=self._system_prompt(),   # 调用约束
        response_format=AgentReviewPayload,    # 结构化输出 Schema
        name="contract_review_agent",
    )
```

### 4.3 系统提示词

[langchain_review.py:246-256](backend/app/services/langchain_review.py#L246-L256)

```text
你是一名合同审查 Agent。
你可以通过 MCP tools 获取合同结构化信息并生成修订条款。
必须遵守以下流程：
1. 先调用 extract_contract_entities 提取合同主体与关键字段；
2. 再调用 risk_assessment_scan 识别风险；
3. 对 HIGH 或 MEDIUM 风险，按需调用 generate_revision_clause 生成替代条款；
4. 在工具结果足够后，再输出最终结构化结论；
5. 禁止在未调用工具时猜测实体、风险或替代条款。
```

### 4.4 Agent 内部循环

当 `agent.ainvoke()` 被调用时，LangGraph 运行时自动执行以下循环：

```text
用户消息："请审查下面的合同内容..."
    │
    ▼
LLM 分析合同文本
    │
    ├── 输出 tool_calls: [extract_contract_entities(text=...)]
    │       │
    │       ▼
    │   框架拦截 → MCP tools/call → MCP Server → 内部调用 LLM → 返回实体 JSON
    │       │
    │       ▼
    │   ToolMessage 回填给 Agent
    │
    ▼
LLM 分析实体结果 + 原文
    │
    ├── 输出 tool_calls: [risk_assessment_scan(text=...)]
    │       │
    │       ▼
    │   框架拦截 → MCP tools/call → MCP Server → 内部调用 LLM → 返回 findings
    │       │
    │       ▼
    │   ToolMessage 回填给 Agent
    │
    ▼
LLM 分析风险结果
    │
    ├── 对每个 HIGH/MEDIUM 风险，输出 tool_calls:
    │   [generate_revision_clause(risk_type=...)]
    │       │
    │       ▼
    │   框架拦截 → MCP tools/call → MCP Server → 内部调用 LLM → 返回条款
    │       │
    │       ▼
    │   ToolMessage 回填给 Agent
    │
    ▼
LLM 拥有所有工具结果
    │
    └── 输出最终 AgentReviewPayload
        (entities + findings + summary)
```

### 4.5 两种工作流模式

| | langchain 模式（Agent 自主编排） | compat_direct 模式（代码硬编码） |
|---|---|---|
| **决策者** | LLM Agent | Python 代码 |
| **调用顺序** | Agent 根据 prompt 约束自行决策 | entities → risks → clauses（固定） |
| **LLM 层次** | 两层（Agent LLM + 工具内部 LLM） | 一层（仅工具内部 LLM） |
| **消息历史** | 完整保留（AI + Tool 消息） | 空 |
| **summary 质量** | LLM 结构化生成 | 硬编码模板 |
| **适用场景** | 生产环境 | 调试、降级、联调 |

## 五、完整执行流程

### 5.1 主流程：合同审查

```text
① 用户在 OnlyOffice 侧边栏输入 contract_id，点击 "开始审查"
   │
   │  POST /api/review/start  {"contract_id": "xxx", "contract_text": "..."}
   │
② routes.py: review_start()
   │  → Pydantic 校验 ReviewStartRequest
   │  → 打开 async DB session
   │  → 调用 run_review(session, contract_id, contract_text)
   │
③ review.py: run_review()
   │  → 从 MySQL 加载 Contract 记录（不存在则 404）
   │  → status = REVIEWING，commit
   │  → 拼接 source_text（优先 contract_text，否则 file_name）
   │  → 调用 _run_review_workflow(source_text)
   │
④ review.py: _run_review_workflow()
   │  → 读取 REVIEW_WORKFLOW_MODE 环境变量
   │  → mode="langchain" → 调用 langchain_workflow.review(text)
   │
⑤ langchain_review.py: review()
   │  → _load_agent() → _load_tools() → 连接 MCP → 创建 Agent
   │  → agent.ainvoke() → Agent 内部循环（见第四章 4.4）
   │  → 解析 structured_response → 收集 tool trace
   │  → 返回 {workflow, entities, findings, summary, messages}
   │
⑥ review.py: run_review() 继续
   │  → 计算 risk_score（HIGH=3, MEDIUM=2, LOW=1 累加）
   │  → 创建 ReviewTask 写入 MySQL
   │  → status = FINISHED，commit
   │  → 返回完整 JSON 响应
   │
⑦ routes.py → FastAPI → 前端
   │  → app.js: renderFindings()
   │  → 渲染风险卡片（红=HIGH / 黄=MEDIUM / 绿=LOW）
   │  → "采纳"按钮 → OnlyOffice ReplaceTextSmart 替换进文档
```

### 5.2 辅助流程：OnlyOffice 文件保存

```text
OnlyOffice Document Server
│  用户在编辑器中保存文档
│  POST /api/onlyoffice/callback
│  {"status": 2, "url": "http://onlyoffice/cache/files/xxx", "key": "doc-key"}
│
① routes.py: onlyoffice_callback()
│  → status == 2 且 url 存在时触发保存
│
② storage.py: save_onlyoffice_file(file_url, key)
│  → 检查 MinIO bucket，不存在则创建
│  → httpx 下载 OnlyOffice 临时文件
│  → MinIO 上传到 onlyoffice/{key}.docx
│  → 返回 minio://contracts/onlyoffice/{key}.docx
│
③ 固定返回 {"error": 0}（OnlyOffice 协议要求）
```

### 6.3 风险评分算法

```python
score = sum(
    3 if severity == "HIGH"
    else 2 if severity == "MEDIUM"
    else 1
    for finding in findings
)
```

示例：2 个 HIGH + 1 个 MEDIUM + 1 个 LOW = 3+3+2+1 = 9.0

## 八、错误处理与日志

### 8.1 异常传播链

```text
MCP 工具内部 LLM 失败
  │  server.py: try/except → 返回降级结果
  │  不会向上抛异常
  │
  ▼
Agent 调用工具失败 / LLM API 错误
  │  langchain_review.py: try/except → 包装为 LangChainWorkflowUnavailable
  │
  ▼
审查工作流失败
  │  review.py: try/except → contract.status = FAILED → HTTPException(500)
  │
  ▼
全局异常处理器 (main.py)
  │  @app.exception_handler(Exception)
  │  → logger.exception() 记录完整堆栈
  │  → 返回 {"detail": "Internal server error: ExceptionType"}
```

### 8.2 各层降级策略

| 层次 | 失败场景 | 降级行为 |
|---|---|---|
| MCP 工具 | LLM 调用超时/报错 | 返回关键词匹配结果或模板文本 |
| Agent | MCP Server 不可用 | `_load_tools()` 抛异常 |
| Agent | LLM 输出无法解析 | 尝试从 messages 中提取 JSON |
| Agent | `ainvoke()` 执行异常 | 包装为 `LangChainWorkflowUnavailable` |
| 审查编排 | 任何工作流异常 | status=FAILED + HTTP 500 |
| 路由层 | 未捕获异常 | 全局处理器返回 500 + JSON |

## 十一、性能特征

### 11.1 延迟分析

Agent 模式下一次完整审查的延迟构成：

```text
总延迟 ≈ Agent LLM 初始化 + Σ(每次工具调用延迟) + 最终总结延迟

每次工具调用延迟 ≈ MCP 传输开销 + 工具内部 LLM 调用延迟
工具内部 LLM 调用延迟 ≈ 2-10 秒（取决于模型和文本长度）

典型场景（3 次工具调用）：
  Agent 初始化: ~1s
  extract_entities: ~3s (MCP 传输 + LLM)
  risk_scan: ~5s (MCP 传输 + LLM)
  revision_clause ×2: ~6s (MCP 传输 + LLM)
  Agent 总结: ~2s
  ─────────────────
  总计: ~17s
```

### 11.2 瓶颈

| 瓶颈 | 原因 | 优化方向 |
|---|---|---|
| LLM 调用延迟 | 每次工具调用内部再调 LLM | 缓存、批处理、更快的模型 |
| 文本截断 | `contract_text[:8000]` 硬截断 | 分段审查、增大上下文窗口 |
| 同步阻塞 | `agent.ainvoke()` 单次阻塞 | 后台队列、异步回调 |
| 串行工具调用 | Agent 逐个调用工具 | 并行工具调用（LangGraph 支持） |
